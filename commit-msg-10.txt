feat(cli): add config-validator docs command

Implements the `docs` command, listing every configuration field
expected by a Schema (RF06), in either a human-readable table (default)
or JSON. Closes #10.

- Supports --format table and --format json; an unknown format raises
  typer.BadParameter with an actionable message.
- Column widths in the table are computed dynamically from the content
  (including headers), so field names/descriptions of any length stay
  aligned without a hardcoded width or an extra table-rendering
  dependency.
- ENUM fields show their accepted values (e.g. "enum (development,
  production)") and LIST fields show their item type (e.g. "lista de
  string") — both formats use the same _field_to_dict/_type_label
  helpers, so table and JSON output never drift apart.
- A field's secret default is masked ("**********") in both output
  formats, consistent with how SecretValue already masks everywhere
  else in the codebase — `docs` exists to expose structure, not secret
  values.

Real finding: adding a second command (`docs`) changes Typer's runtime
behavior — a Typer() app with only ONE registered command "collapses"
and skips requiring the subcommand name (as check was, alone, in #9).
With two commands, `check` and `docs` are both required explicitly again
(`config-validator check ...` / `config-validator docs ...`, matching
the original RF09/RF06 wording). Updated test_check_command.py's
CliRunner invocations accordingly.

Also updated docs/architecture.md: marked the SecretValue-at-runtime
debt (item 1) as resolved (it was closed back in #7/#8 but the doc
still said "pending"), kept the namespace_tree-as-dict decision (item 2)
as still valid, and logged a new roadmap item (--format markdown for
docs, candidate for M5).

Tests: 12 new tests (test_docs_command.py), 149 total across the
project, 100% coverage (branch included, including ENUM/LIST branches
in _type_label). Lint: ruff clean. Type-check: mypy --strict clean on
src/.
